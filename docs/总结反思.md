# 家账项目总结反思

## 项目概述

用 React 19 + Supabase 搭了一套家庭记账 Web 应用。多成员各自登录、数据按家庭聚合、月末自动统计。全程零成本：Supabase Free + Cloudflare Pages 免费额度。

---

## 技术栈

| 层 | 选型 |
|---|---|
| 框架 | React 19 + Vite + TypeScript |
| 样式 | Tailwind CSS 4 + shadcn/ui |
| 路由 | React Router 7 |
| 服务端状态 | TanStack Query 5 |
| 后端 | Supabase（Auth + PostgreSQL + RLS） |
| 图表 | Recharts |
| 部署 | Cloudflare Pages |

---

## 构建过程

### 初始化

Vite 脚手架搭起项目，安装 Tailwind 4 + shadcn/ui + React Router + TanStack Query + Supabase JS + Recharts + date-fns + Lucide。配置了 `@/` 路径别名、Prettier、Tailwind 插件、shadcn components.json。按 features 维度划分目录结构。

### 业务代码

所有功能模块集中在 **4 个 features**：

- **auth**：登录注册自动检测，AuthContext + AuthProvider 封装
- **family**：创建家庭（生成邀请码）、通过邀请码加入、成员列表
- **transactions**：快捷记账（类别 + 金额 + 支付方式 + 备注 + 标签）、账单列表、删除
- **stats**：月度饼图、成员排名、月份切换

Supabase 客户端统一在 `lib/supabase.ts` 初始化，环境变量缺失时降级为 null。类型定义手动维护，与 DB 列名一致（snake_case）。

### 数据库

4 张表：`profiles`（用户扩展）、`families`（家庭）、`family_members`（成员关系）、`transactions`（账单）。用 Supabase 自动注册触发器创建 profile。

## 部署历程

### Vercel（已弃用）

第一次部署到 Vercel Hobby。设置了 SPA fallback `vercel.json`。访问正常，但 `vercel.app` 域名被 GFW 屏蔽，国内无法使用。放弃 Vercel。

### Cloudflare Pages（当前）

迁移到 Cloudflare Pages。国内访问速度尚可，免费额度对小型应用足够。

构建产物直接上传（`wrangler pages deploy dist --project-name jiazhang`），没有连接 Git 仓库自动部署——每次代码变更后手动跑 `pnpm build && wrangler pages deploy`。

SPA 路由通过 `public/_redirects` 文件处理：`/* /index.html 200`。

## 踩坑记录

### 1. RLS 无限递归

**错误**：创建家庭时 `infinite recursion detected in policy for relation "family_members"`

**原因**：`family_members` 表的 SELECT 策略中又查询了同一张表：

```sql
create policy "members can read own family members"
  on public.family_members for select
  using (
    family_id in (
      select family_id from public.family_members where user_id = auth.uid()
    )
  );
```

Supabase 的 RLS 策略在执行时，子查询也会被策略拦截，导致层层嵌套无法终止。

**解决**：创建一个 `security definer` 函数，函数内部的查询绕过 RLS：

```sql
create or replace function public.is_family_member(family_id uuid)
returns boolean
language sql
security definer
stable
as $$
  select exists (
    select 1 from public.family_members
    where family_members.family_id = $1
    and family_members.user_id = auth.uid()
  );
$$;
```

所有涉及家庭成员身份检查的策略改用该函数。`security definer` 意味着以函数所有者权限执行，不受调用者 RLS 影响。

### 2. 缺少 INSERT 策略

**错误**：创建家庭时 `new row violates row-level security policy for table "families"`

**原因**：数据库有 SELECT 策略但没有 INSERT 策略。另外，创建家庭后 `.select()` 返回数据时会被 SELECT 策略拦截（用户尚未加入 family_members，无法通过家庭成员检查）。

**解决**：

```
families     → 新增 INSERT 策略（允许已登录用户创建）
families     → 放宽 SELECT 策略（允许已登录用户读取）
family_members → 新增 INSERT 策略（user_id = auth.uid()，只能添加自己）
```

### 3. Vercel 被墙

**问题**：`jiazhang-six.vercel.app` 在国内无法访问。全家要用，必须解决。

**方案**：改用 Cloudflare Pages。CF 国内节点多，访问快，有免费额度。

### 4. Supabase 邮箱注册限制

**问题**：用户在 Cloudflare Pages 上注册时提示 "email rate limit exceeded"。Supabase Free 默认 `rate_limit_email_sent = 2`，同一个邮箱每小时只能发 2 封邮件。

**解决**：在 Supabase Auth 设置中开启 `mailer_autoconfirm: true`，注册后自动确认，不再发确认邮件，绕过了发送频率限制。适合家庭场景，不需要确认邮件验证。

## 经验教训

### RLS 策略设计

Supabase 的 RLS 策略在查询当前表时容易踩递归坑。设计原则：

1. 不在同一个表的 SELECT 策略中查询该表自身
2. 如果必须查，用 `security definer` 函数隔一层
3. INSERT 策略要和 SELECT 策略配套设计，考虑 `insert + select` 组合场景

### 部署选型

面向国内用户的服务，部署在国内 CDN 有覆盖的平台是刚需。Cloudflare Pages 免费且支持中国节点，比 Vercel 更适合此类场景。Git 自动部署还没接，手动两步构建可行但繁琐，后续可以接。

### 监控缺失

没有错误监控。用户只反馈了 "创建不了"，定位全靠猜测和查数据库。对于小工具项目可以接受，但从第一天接入 Sentry 或类似服务也不费事。

### 状态管理

用了 TanStack Query 管理服务端状态，客户端只用 React `useState`，没有引入 Redux/Zustand。这个选择是对的——没有跨组件复杂状态，Query 的缓存和失效机制足够。

### 类型安全

手动维护 TypeScript 类型（与 DB 列名对应）在小项目里可行，但表结构有变更时容易遗漏。应该用 `supabase gen types typescript --linked` 自动生成。
